# SPEC-005 Tasks - Workflow State

## TASK 005.1 - Workflow State Schemas and Lifecycle

### Objective

Define typed workflow state schemas and lifecycle status values aligned with
`SPEC.md` and project contracts.

### Scope

- Add Pydantic schemas for WorkflowState and nested state sections.
- Define workflow status enum values.
- Define terminal status helpers where useful.
- Keep schemas generic enough for future workflow domains.

### Deliverables

- Workflow state schema module.
- Workflow lifecycle/status definitions.
- Unit tests for schema parsing, serialization, and status values.
- README notes for workflow state foundation if useful.

### Acceptance Criteria

- WorkflowState can represent the state shape from `SPEC.md`.
- Workflow statuses match the approved status list.
- Terminal statuses are identifiable.
- Tests prove valid schema parsing and invalid payload rejection.

### Out-of-scope

- Database migrations.
- API endpoints.
- LangGraph runtime.
- Agents.
- Procurement-specific Agent outputs.

## TASK 005.2 - Workflow Transition Rules

### Objective

Implement explicit workflow status transition validation.

### Scope

- Define allowed workflow status transitions.
- Reject invalid transitions.
- Reject transitions from terminal statuses.
- Add typed exceptions for lifecycle errors where useful.

### Deliverables

- Workflow transition rule module.
- Tests for accepted transitions.
- Tests for rejected transitions and terminal status behavior.

### Acceptance Criteria

- `CREATED -> PLANNING` is accepted.
- direct `CREATED -> COMPLETED` is rejected.
- terminal statuses cannot transition.
- failure and cancellation transitions follow the SPEC-005 lifecycle rules.
